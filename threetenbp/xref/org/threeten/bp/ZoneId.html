57  /**
58   * A time-zone ID, such as {@code Europe/Paris}.
59   * <p>
60   * A {@code ZoneId} is used to identify the rules used to convert between
61   * an {@link Instant} and a {@link LocalDateTime}.
62   * There are two distinct types of ID:
63   * <ul>
64   * <li>Fixed offsets - a fully resolved offset from UTC/Greenwich, that uses
65   *  the same offset for all local date-times
66   * <li>Geographical regions - an area where a specific set of rules for finding
67   *  the offset from UTC/Greenwich apply
68   * </ul>
69   * Most fixed offsets are represented by {@link ZoneOffset}.
70   * Calling {@link #normalized()} on any {@code ZoneId} will ensure that a
71   * fixed offset ID will be represented as a {@code ZoneOffset}.
72   * <p>
73   * The actual rules, describing when and how the offset changes, are defined by {@link ZoneRules}.
74   * This class is simply an ID used to obtain the underlying rules.
75   * This approach is taken because rules are defined by governments and change
76   * frequently, whereas the ID is stable.
77   * <p>
78   * The distinction has other effects. Serializing the {@code ZoneId} will only send
79   * the ID, whereas serializing the rules sends the entire data set.
80   * Similarly, a comparison of two IDs only examines the ID, whereas
81   * a comparison of two rules examines the entire data set.
82   *
83   * <h3>Time-zone IDs</h3>
84   * The ID is unique within the system.
85   * There are three types of ID.
86   * <p>
87   * The simplest type of ID is that from {@code ZoneOffset}.
88   * This consists of 'Z' and IDs starting with '+' or '-'.
89   * <p>
90   * The next type of ID are offset-style IDs with some form of prefix,
91   * such as 'GMT+2' or 'UTC+01:00'.
92   * The recognised prefixes are 'UTC', 'GMT' and 'UT'.
93   * The offset is the suffix and will be normalized during creation.
94   * These IDs can be normalized to a {@code ZoneOffset} using {@code normalized()}.
95   * <p>
96   * The third type of ID are region-based IDs. A region-based ID must be of
97   * two or more characters, and not start with 'UTC', 'GMT', 'UT' '+' or '-'.
98   * Region-based IDs are defined by configuration, see {@link ZoneRulesProvider}.
99   * The configuration focuses on providing the lookup from the ID to the
100  * underlying {@code ZoneRules}.
101  * <p>
102  * Time-zone rules are defined by governments and change frequently.
103  * There are a number of organizations, known here as groups, that monitor
104  * time-zone changes and collate them.
105  * The default group is the IANA Time Zone Database (TZDB).
106  * Other organizations include IATA (the airline industry body) and Microsoft.
107  * <p>
108  * Each group defines its own format for the region ID it provides.
109  * The TZDB group defines IDs such as 'Europe/London' or 'America/New_York'.
110  * TZDB IDs take precedence over other groups.
111  * <p>
112  * It is strongly recommended that the group name is included in all IDs supplied by
113  * groups other than TZDB to avoid conflicts. For example, IATA airline time-zone
114  * region IDs are typically the same as the three letter airport code.
115  * However, the airport of Utrecht has the code 'UTC', which is obviously a conflict.
116  * The recommended format for region IDs from groups other than TZDB is 'group~region'.
117  * Thus if IATA data were defined, Utrecht airport would be 'IATA~UTC'.
118  *
119  * <h3>Serialization</h3>
120  * This class can be serialized and stores the string zone ID in the external form.
121  * The {@code ZoneOffset} subclass uses a dedicated format that only stores the
122  * offset from UTC/Greenwich.
123  * <p>
124  * A {@code ZoneId} can be deserialized in a Java Runtime where the ID is unknown.
125  * For example, if a server-side Java Runtime has been updated with a new zone ID, but
126  * the client-side Java Runtime has not been updated. In this case, the {@code ZoneId}
127  * object will exist, and can be queried using {@code getId}, {@code equals},
128  * {@code hashCode}, {@code toString}, {@code getDisplayName} and {@code normalized}.
129  * However, any call to {@code getRules} will fail with {@code ZoneRulesException}.
130  * This approach is designed to allow a {@link ZonedDateTime} to be loaded and
131  * queried, but not modified, on a Java Runtime with incomplete time-zone information.
132  *
133  * <h3>Specification for implementors</h3>
134  * This abstract class has two implementations, both of which are immutable and thread-safe.
135  * One implementation models region-based IDs, the other is {@code ZoneOffset} modelling
136  * offset-based IDs. This difference is visible in serialization.
137  */
138 public abstract class ZoneId implements Serializable {

288     /**
289      * Obtains an instance of {@code ZoneId} from an ID ensuring that the
290      * ID is valid and available for use.
291      * <p>
292      * This method parses the ID producing a {@code ZoneId} or {@code ZoneOffset}.
293      * A {@code ZoneOffset} is returned if the ID is 'Z', or starts with '+' or '-'.
294      * The result will always be a valid ID for which {@link ZoneRules} can be obtained.
295      * <p>
296      * Parsing matches the zone ID step by step as follows.
297      * <ul>
298      * <li>If the zone ID equals 'Z', the result is {@code ZoneOffset.UTC}.
299      * <li>If the zone ID consists of a single letter, the zone ID is invalid
300      *  and {@code DateTimeException} is thrown.
301      * <li>If the zone ID starts with '+' or '-', the ID is parsed as a
302      *  {@code ZoneOffset} using {@link ZoneOffset#of(String)}.
303      * <li>If the zone ID equals 'GMT', 'UTC' or 'UT' then the result is a {@code ZoneId}
304      *  with the same ID and rules equivalent to {@code ZoneOffset.UTC}.
305      * <li>If the zone ID starts with 'UTC+', 'UTC-', 'GMT+', 'GMT-', 'UT+' or 'UT-'
306      *  then the ID is a prefixed offset-based ID. The ID is split in two, with
307      *  a two or three letter prefix and a suffix starting with the sign.
308      *  The suffix is parsed as a {@link ZoneOffset#of(String) ZoneOffset}.
309      *  The result will be a {@code ZoneId} with the specified UTC/GMT/UT prefix
310      *  and the normalized offset ID as per {@link ZoneOffset#getId()}.
311      *  The rules of the returned {@code ZoneId} will be equivalent to the
312      *  parsed {@code ZoneOffset}.
313      * <li>All other IDs are parsed as region-based zone IDs. Region IDs must
314      *  match the regular expression <code>[A-Za-z][A-Za-z0-9~/._+-]+</code>
315      *  otherwise a {@code DateTimeException} is thrown. If the zone ID is not
316      *  in the configured set of IDs, {@code ZoneRulesException} is thrown.
317      *  The detailed format of the region ID depends on the group supplying the data.
318      *  The default set of data is supplied by the IANA Time Zone Database (TZDB).
319      *  This has region IDs of the form '{area}/{city}', such as 'Europe/Paris' or 'America/New_York'.
320      *  This is compatible with most IDs from {@link java.util.TimeZone}.
321      * </ul>
322      *
323      * @param zoneId  the time-zone ID, not null
324      * @return the zone ID, not null
325      * @throws DateTimeException if the zone ID has an invalid format
326      * @throws ZoneRulesException if the zone ID is a region ID that cannot be found
327      */
328     public static ZoneId of(String zoneId) {
329         Objects.requireNonNull(zoneId, "zoneId");
330         if (zoneId.equals("Z")) {
331             return ZoneOffset.UTC;
332         }
333         if (zoneId.length() == 1) {
334             throw new DateTimeException("Invalid zone: " + zoneId);
335         }
336         if (zoneId.startsWith("+") || zoneId.startsWith("-")) {
337             return ZoneOffset.of(zoneId);
338         }
339         if (zoneId.equals("UTC") || zoneId.equals("GMT") || zoneId.equals("UT")) {
340             return new ZoneRegion(zoneId, ZoneOffset.UTC.getRules());
341         }
342         if (zoneId.startsWith("UTC+") || zoneId.startsWith("GMT+") ||
343                 zoneId.startsWith("UTC-") || zoneId.startsWith("GMT-")) {
344             ZoneOffset offset = ZoneOffset.of(zoneId.substring(3));
345             if (offset.getTotalSeconds() == 0) {
346                 return new ZoneRegion(zoneId.substring(0, 3), offset.getRules());
347             }
348             return new ZoneRegion(zoneId.substring(0, 3) + offset.getId(), offset.getRules());
349         }
350         if (zoneId.startsWith("UT+") || zoneId.startsWith("UT-")) {
351             ZoneOffset offset = ZoneOffset.of(zoneId.substring(2));
352             if (offset.getTotalSeconds() == 0) {
353                 return new ZoneRegion("UT", offset.getRules());
354             }
355             return new ZoneRegion("UT" + offset.getId(), offset.getRules());
356         }
357         return ZoneRegion.ofId(zoneId, true);
358     }

360     /**
361      * Obtains an instance of {@code ZoneId} wrapping an offset.
362      * <p>
363      * If the prefix is "GMT", "UTC", or "UT" a {@code ZoneId}
364      * with the prefix and the non-zero offset is returned.
365      * If the prefix is empty {@code ""} the {@code ZoneOffset} is returned.
366      *
367      * @param prefix  the time-zone ID, not null
368      * @param offset  the offset, not null
369      * @return the zone ID, not null
370      * @throws IllegalArgumentException if the prefix is not one of
371      *     "GMT", "UTC", or "UT", or ""
372      */
373     public static ZoneId ofOffset(String prefix, ZoneOffset offset) {
374         Objects.requireNonNull(prefix, "prefix");
375         Objects.requireNonNull(offset, "offset");
376         if (prefix.length() == 0) {
377             return offset;
378         }
379         if (prefix.equals("GMT") || prefix.equals("UTC") || prefix.equals("UT")) {
380             if (offset.getTotalSeconds() == 0) {
381                 return new ZoneRegion(prefix, offset.getRules());
382             }
383             return new ZoneRegion(prefix + offset.getId(), offset.getRules());
384         }
385         throw new IllegalArgumentException("Invalid prefix, must be GMT, UTC or UT: " + prefix);
386     }

494     /**
495      * Normalizes the time-zone ID, returning a {@code ZoneOffset} where possible.
496      * <p>
497      * The returns a normalized {@code ZoneId} that can be used in place of this ID.
498      * The result will have {@code ZoneRules} equivalent to those returned by this object,
499      * however the ID returned by {@code getId()} may be different.
500      * <p>
501      * The normalization checks if the rules of this {@code ZoneId} have a fixed offset.
502      * If they do, then the {@code ZoneOffset} equal to that offset is returned.
503      * Otherwise {@code this} is returned.
504      *
505      * @return the time-zone unique ID, not null
506      */
507     public ZoneId normalized() {
508         try {
509             ZoneRules rules = getRules();
510             if (rules.isFixedOffset()) {
511                 return rules.getOffset(Instant.EPOCH);
512             }
513         } catch (ZoneRulesException ex) {
514             // ignore invalid objects
515         }
516         return this;
517     }
